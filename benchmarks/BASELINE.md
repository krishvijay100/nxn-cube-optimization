Captured 2026-07-11 on Apple Silicon (14-core, L1D 64 KiB, L2 4 MiB per cluster).
Build: `-DCMAKE_BUILD_TYPE=Release` (default), Apple Clang, single-threaded solver.

### Theoretical unpruned tree size (baseline for comparison)

Deriving what an unpruned IDA* search would visit at the same solve depths:

- Phase 1: 18 raw moves, ~1/3 removed by redundancy pruning (same-face + parallel-face suppression) → effective branching ≈ **13.3**. Avg Phase 1 depth ≈ 9.5. Tree ≈ `13.3^9.5 ≈ 2.4×10¹⁰`.
- Phase 2: 10 raw G1 moves, similar redundancy → effective branching ≈ **7**. Avg Phase 2 depth ≈ 13.7. Tree ≈ `7^13.7 ≈ 4.7×10¹¹`.
- **Combined theoretical unpruned tree: ~5×10¹¹ nodes** (Phase 2 dominates).

### Collapse ratio

Measured ~278k explored vs ~5×10¹¹ theoretical unpruned → **~1.8×10⁶× reduction** in nodes visited (equivalently, **>99.99994%** of the theoretical tree pruned).

Phase 2 dominates both the *time* budget (98% of serial latency, from below) and the *space* budget (95% of nodes explored). Pruning-table quality on Phase 2 is the single lever with the highest expected value for future perf work.

## Rejected permanently — measured twice, doesn't help in this codebase

- **Fixed-size `Path` (std::array<uint8_t, 32> + size counter, replacing std::vector<int>).** Regressed BM_Solve by ~6% on both attempts:
  - **First pass (against Ctx-only baseline):** 1.31 → 1.39 ms (-6%).
  - **Retry on top of bitboard-packed state:** 1.31 → 1.40 ms (-6%).
  - The bitboard change was expected to free cache-line budget for the 33-byte stack buffer, but the regression was identical, so that hypothesis was falsified. Root cause stands: libc++'s `std::vector<int>` `push_back`/`pop_back` is already inlining to essentially optimal code under `-O3`, and the fixed-size stack object costs more than it saves. Not retrying again.

## Rejected — symmetry-compressed pruning tables (built, verified, measured, didn't help)

Built a full 8-element Phase-2 symmetry subgroup (generated by S_U2, S_F2, S_LR2 — the commuting order-2 rotations/reflections that preserve the G1 move set). Verified correctness comprehensively (11 new tests covering group closure, inverses, solved-cube invariance, conjugation-commutes-with-moves at both cube-state and coord level, and compressed-vs-raw exact-match on all 967,680 pairs of each table). Wired the compressed lookup into `phase2_h`.

- **Compression ratio achieved:** ~7.78× on both `(cp, sp)` and `(ep, sp)` tables. Distance tables shrank from ~945 KB → ~124 KB each.
- **BM_Solve result:** 1.31 ms → **1.62 ms mean (-24%)**. All solutions still correct (avg length 23.17 unchanged, 116/116 tests pass).
- **Root cause of the regression:** the `class_of[main][sp]` index table needed to map every raw pair to its class costs 4 bytes per entry (`uint32_t` — 120K+ classes overflow `uint16_t`). At `40320 × 24 × 4 B ≈ 3.87 MB` per side, `class_of` is *~4× larger than the raw pruning table it replaces*, and it lives outside L2 for large sweeps of the DFS. The extra indirection eats the L2-hit savings of the smaller compressed distance table, and then some.
- **Deeper issue:** the reachable ceiling of the Phase-2-preserving symmetry subgroup we can build without breaking G1 (using only commuting order-2 generators) is 8 elements, not 16 or 48. Kociemba-literature "48× compression" numbers rely on either full Oₕ + a symmetry-aware search that handles non-G1-preserving conjugates, or on a fundamentally different heuristic layout (e.g., corner-perm-alone without slice, then combined post-hoc). Neither is a small change.
- **What would still be worth trying (out of scope for this pass):** a denser single-table heuristic (e.g., `(cp, ep, sp)` jointly, currently untractable at ~40K×40K×24 ≈ 39 GB raw but manageable at 39 GB / ~48 ≈ 800 MB post-compression — still too big) or symmetry compression on a *single* coord dimension where the `class_of` overhead is small compared to the raw dimension.

Reverted the wire-in. Kept `symmetry.h/cpp` + tests as a documented artifact — the infrastructure is correct and could be reused for a future denser-heuristic experiment. Full details of the correctness proof live in `tests/symmetry_test.cpp` and `tests/sym_pruning_test.cpp`.

near-linear scaling through the physical core count. per-thread latency
barely moves as threads grow — the solver is not fighting any shared state.

## Where the time goes

Phase 2 is ~98% of total solve time (2.08 ms out of 2.12 ms). Any optimization
to phase 2 dominates the overall speedup. Phase 1 is essentially free — its
pruning table is tight and it finds short solutions (9.5 moves avg) quickly.
